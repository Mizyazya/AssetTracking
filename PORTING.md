# PORTING.md — специфікація паритету Assets v1 → v2

Еталон поведінки — гілка `v1` (файл `app.py`, тег `v1.5-final`). Цей документ — повний опис того, **що** має робити v2, щоб замінити v1 у продакшні. Імплементація за цим документом не потребує читання v1-коду, але у спірних випадках еталон — `git show v1:app.py`.

**Правило для імплементатора:** усе, що тут не позначено як «Відхилення від v1», портується поведінково один-в-один, включно з текстами повідомлень українською.

## 2. Ledger компонентів — серце системи

Для `type='component'` **немає таблиці поточних власників**. Хто скільки тримає — завжди обчислюється реплеєм `asset_log` цього активу в порядку `timestamp`:

```
balance[person_id] += quantity  якщо action == 'Видано'
balance[person_id] -= quantity  якщо action ∈ ('Повернення', 'Повернено')
(NULL quantity рахується як 0; person_id NULL ігнорується)
держателі = {person_id: qty | qty > 0}
```

У v1 цей цикл скопійований у 6 місцях. **Відхилення від v1 (структурне):** у v2 — ЄДИНИЙ модуль `src/lib/ledger.ts` з функціями на кшталт `componentHolders(assetId)`, `holderBalance(assetId, personId)`, `holdersForPeople(assetIds, personIds)`. Поведінка ідентична.

Повний словник action-рядків (сентинелі, порівняння точним збігом; НЕ перекладати, НЕ перейменовувати — вони в історичних даних):
- `'Створено'` — початковий запис (quantity: 1 для активу, N для компонента)
- `'Поставка'` — поповнення складу компонента (+quantity до asset.quantity)
- `'Видано'` — компонент видано особі (+ у ledger)
- `'Повернення'` — компонент повернено від особи (− у ledger)
- `'Повернено'` — актив повернено на склад / історичний синонім «Повернення» у ledger
- `'Передано'` / `'Отримано'` — пара записів при передачі активу між особами (на ledger не впливають)
- `'Задача'` / `'Задача закрита'` — події задач (на ledger не впливають)

Нові quantity-впливові action-и додавати ТІЛЬКИ через ledger.ts.

## 3. Автентифікація і сесії

- Логін: username **без урахування регістру** (`lower(username)`), пароль — див. §5. Помилка: `'Невірний логін або пароль'`.
- При вході: створюється `user_session` (uuid4 → `session_id`, ip, user_agent, active=1), `session_id` кладеться у підписану cookie разом з user id.
- **Ковзний строк життя сесії — 30 хвилин**: кожен автентифікований запит подовжує cookie на 30 хв.
- **Перевірка на кожному запиті** (middleware): якщо в cookie немає sid, або `user_session` з цим sid не існує чи `active=0` — примусовий вихід, редірект на /login з повідомленням `'Сесію завершено адміністратором або з іншого пристрою.'` (при цьому якщо запис існує — виставити active=0, logout_time).
- Вихід (/logout): active=0, logout_time=now, очистити cookie.
- Завершення сесії (`close_session`): адмін — будь-яку, користувач — лише свою; active=0 + logout_time. Flash `'Сесію завершено'`.
- **Захист маршрутів:** УСІ сторінки і дії вимагають входу, крім /login і статики. Адмінські: /users, /add_user, /edit_user, /delete_user, /user_sessions/<id> — інакше 403.
- Ролі: `admin`, `user`. Правила:
  - користувач не може змінити **власну** роль (навіть адмін собі);
  - не можна видалити останнього адміна: `'Неможливо видалити останнього адміністратора!'`;
  - зміна пароля собі — /profile (доступно всім ролям).
- Бутстрап: якщо таблиця user порожня — створити `admin`/`admin` (role=admin) і повідомити в лог про необхідність зміни пароля.
- Cookie підписується `SECRET_KEY` з env.

## 4. Валідація пароля (add_user, edit_user, profile)

8–64 символи, обов'язково: велика літера, мала літера, цифра, не-алфанумерний символ. Помилка: `'Пароль не відповідає вимогам безпеки.'`. У edit_user порожній пароль = «не змінювати».

## 5. Werkzeug-сумісні паролі (критично для міграції)

Існуючі хеші мають формат werkzeug `method$salt$hexhash`:
- `pbkdf2:sha256:<iterations>$<salt>$<hex>` — crypto.pbkdf2Sync(password, salt, iterations, 32, 'sha256')
- `scrypt:<N>:<r>:<p>$<salt>$<hex>` — crypto.scryptSync(password, salt, 64, {N, r, p, maxmem: 0x7fffffff})

Порівняння — timingSafeEqual. Нові/оновлені паролі v2 хешує власним форматом на node:crypto scrypt (без нативних залежностей), напр. `scrypt$N$r$p$saltB64$hashB64`. При успішному вході зі старим werkzeug-хешем — прозоро пере-хешувати у формат v2. Модуль: `src/lib/password.ts`.

## 6. Маршрути і поведінка

Мапінг v1 → v2 (App Router; POST-дії — server actions при тих самих сторінках):

| v1 | v2 сторінка | поведінка |
|---|---|---|
| GET / | `/` | список майна, див. §7.1 |
| GET /asset/<id> | `/assets/[id]` | деталі: історія (пагінація 15, новіші згори), задачі (тільки active-активи), держателі; форми видачі/повернення/поставки/задачі/редагування |
| GET /people | `/people` | §7.2 |
| GET /person/<id> | `/people/[id]` | активи особи, компоненти через ledger, історія (15/стор.) |
| GET /locations | `/locations` | список + додавання/редагування/видалення |
| GET /location/<id> | `/locations/[id]` | люди локації, їх активи, компоненти через ledger по кожній особі |
| GET /tasks | `/tasks` | §7.3 |
| GET /users, /add_user, /edit_user | `/users`, `/users/new`, `/users/[id]` | адмінка |
| GET /user_sessions/<id>, /my_sessions | `/users/[id]/sessions`, `/profile/sessions` | список сесій, кнопка завершення активних |
| GET/POST /profile | `/profile` | зміна власного пароля |
| GET/POST /login, /logout | `/login`, дія logout | |
| POST /backup_database | дія на будь-якій сторінці (кнопка у футері) | копія SQLite-файлу → `backup/assets_backup_YYYYMMDD_HHMMSS.db`; flash успіх/помилка. **Відхилення від v1:** дія тільки для адмінів (у v1 кнопка була адмінська, а маршрут — ні) |
| /register | — | **НЕ портується** (мертвий у v1: шаблону не існувало) |

Дії (server actions), поведінка 1:1 з v1:

- **add_asset**: `active` → quantity=1, serial з форми або NULL; дубль серійника → `'Актив з таким серійним номером вже існує'`. `component` → quantity з форми (default 1), порожній serial → автогенерація `component-<unix_ms>`; дубль → `'Компонент з таким серійним номером вже існує'`. Лог `'Створено'` + `'Майно додано'`.
- **assign_asset (актив)**: current_holder_id=person, status=`'У користуванні'`; два логи: попередньому держателю `'Передано'` (якщо був), новому `'Отримано'`, з іменами і коментарем (формати рядків — з v1).
- **assign_asset (компонент, зі складу)**: якщо asset.quantity ≥ qty → мінус зі складу, лог `'Видано'` (person_id=отримувач); інакше flash `'Недостатньо компонентів на складі'`.
- **assign_asset (компонент, від особи до особи)**: перевірка балансу віддавача через ledger; недостатньо → `'Недостатньо компонентів у користувача для передачі'`; інакше пара логів `'Повернення'`(від) + `'Видано'`(кому). Склад НЕ змінюється.
- **return_asset (актив)**: holder=NULL, status=`'На складі'`, лог `'Повернено'`.
- **return_asset (компонент)**: перевірка балансу особи; недостатньо → `'Недостатньо компонентів у користувача для повернення'`; інакше asset.quantity += qty, лог `'Повернення'`.
- **add_component_supply**: asset.quantity += qty, лог `'Поставка'` (коментар з форми або `'Поставка компоненту'`).
- **add_task**: тільки непорожній текст; Task + лог `'Задача'` з текстом.
- **close_task**: якщо вже закрита → flash `'Задача вже закрита.'`; інакше closed_at=now, close_comment, лог `'Задача закрита'`, flash `'Задачу закрито.'`.
- **edit_asset**: name/serial/comments; `'Дані активу оновлено'`.
- **add_person**: непорожнє ім'я (`'Імʼя не може бути порожнім'`), опційна локація; `'Людину додано'`.
- **edit_person**: ім'я, телефон (порожній → NULL), локація; `'Дані людини оновлено'`.
- **delete_person**: заборонено, якщо за особою є активи (current_holder_id) — `'Не можна видалити людину, за якою закріплене майно'`. **Відхилення від v1 (фікс):** також заборонено при ненульовому балансі компонентів у ledger (v1 це обіцяв у README, але не перевіряв).
- **add/edit/delete_location**: назва непорожня (`'Назва локації не може бути порожньою'`); видалення локації відв'язує людей (location_id=NULL), НЕ видаляє їх.
- **add_user / edit_user**: унікальність логіна без регістру (`'Користувач з таким логіном вже існує'`); валідація пароля §4; заборона зміни власної ролі.
- **delete_user**: захист останнього адміна; `'Користувача видалено'`.

## 7. Списки: фільтри, сортування, пагінація

Скрізь **15 рядків на сторінку**. Фільтри передаються query-параметрами і зберігаються при переході по сторінках пагінації та сортуванні.

### 7.1 Головна (список майна)
- Фільтри: name (ilike %..%), serial (ilike), comments (ilike), type (точно), location_id, person_id.
- **Нюанс location_id/person_id:** окрім активів із відповідним current_holder_id, у результат включаються компоненти, у яких **ledger-баланс > 0** у відповідної особи / будь-якої особи локації.
- Сортування: name, created_at (default, desc), task_count (кількість активних задач), location (назва локації держателя, порожня — перша). Клік по колонці перемикає asc/desc.
- Для кожного рядка: держатель (актив → особа+локація; компонент → список (особа, кількість) з ledger).

### 7.2 Люди
- Фільтри: location_id (спецзначення `none` = без локації), search по імені (ilike), has_assets: `yes` = має активи АБО компоненти, `no` = не має нічого. Опис активних фільтрів показується рядком (`'Локація: X'`, `'Пошук: 'Y''`, `'Має майно'`/`'Немає майна'`).
- Сортування за іменем, пагінація 15.
- **Нюанс v1:** has_assets перевіряє тільки `Asset.current_holder_id` (relationship). Компонентні баланси НЕ враховуються. Портувати як є.

### 7.3 Задачі
- Дві секції: активні (сортування created_at asc, згруповані по активу) і закриті (closed_at desc). Обидві пагіновані по 15 (спільний параметр page — як у v1).
- Фільтри (застосовуються до обох секцій): location_id (по локації держателя активу), person_id (по держателю), asset_name (ilike), task_text (ilike), date_from/date_to (по created_at, включно, to = кінець дня), closed_from/closed_to (тільки закриті).
- Тільки задачі активів (`asset.type='active'`).

## 8. Час і часові пояси

**Відхилення від v1 (нормалізація):** у v1 мітки часу — суміш naive UTC (`asset.created_at`, `user_session.login_time`) і локального Києва (`asset_log.timestamp`, `task.created_at`). У v2:
- у БД усе зберігається як **UTC ISO-рядок**;
- відображення — завжди у `TIMEZONE` з env (default `Europe/Kyiv`), формат як у v1-шаблонах (`YYYY-MM-DD HH:MM`);
- скрипт імпорту конвертує старі київські мітки (`asset_log.timestamp`, `task.created_at/closed_at`) у UTC, а naive-UTC поля переносить як є (див. scripts/import-v1.ts).

## 9. Env і конфігурація

`.env`: `SECRET_KEY` (обов'язковий у продакшні; без нього — гучне попередження і dev-фолбек), `TIMEZONE` (default `Europe/Kyiv`). Шлях БД — константа `data/assets.db` (не env). Папки `data/` і `backup/` створюються автоматично при старті — це фікс v1-болю «перший запуск падає без instance/».

## 10. Відомі особливості v1, свідомо НЕ портовані

1. `/register` — мертвий маршрут без шаблону (500 при відкритті).
2. Падіння першого запуску без папки `instance/`.
3. Неадмінський доступ до POST /backup_database.
4. delete_person не враховував компонентні баланси.
5. PostgreSQL-гілки коду і pg_dump-бекап.
6. 6 копій ledger-циклу.

## 11. Мова інтерфейсу

Весь UI — українською, тексти flash-повідомлень і action-рядків — точно як у цьому документі (вони збігаються з v1). Нові тексти — у тому ж стилі.
